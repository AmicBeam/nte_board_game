# NTE Board Game Architecture

## 当前目标

本项目是一个独立的 Flask 网站，提供异环主题的网页卡牌牌桌，并通过数据库保存玩家、登录态、构筑和对局状态。

当前主游戏模式是“异象对决”：玩家使用异象道具作为主牌组，通过启动、调度、回收和小干扰争夺资源；异能者作为异能者编队，满足复杂素材条件后登场或再次共鸣，并立刻带来铺场、清场、返场、资源中转或终端爆发。

异象对决的现行规则、术语、卡牌设计和异能者职责必须遵循 `docs/everness-item-chain-card-design.md`。该文件是当前唯一设计真源；`docs/duel-balance-iteration.md` 只保留测试流程、评分标准和迭代报告索引。

历史探索玩法已从当前代码与资源中移除，不保留兼容入口。新增规则、文档和 UI 必须围绕卡牌对决展开。

与机器人交互的代码位于 `plugins/` 目录；网页端只负责展示状态和发送用户操作，所有规则计算都在后端完成。

## 设计宗旨

游戏应同时满足轻度游玩、逻辑挑战、随机刺激和强成就反馈。每次更新都应优先降低玩家记忆负担和操作负担，让玩家能快速理解当前局势、做出选择并看到结果；同时通过清晰的规则关系、可预判的风险收益和逐步抬升的难度，保留足够推理空间与挑战感。

异象对决的核心体验是：

- 起手拿到动点和小干扰。
- 1 到 2 费围绕展开件、解对手展开件和保素材做取舍。
- 3 费开始尝试登场第一名异能者，对手应有机会通过费用、战力、素材和手牌干扰阻止。
- 5 到 6 费登场王牌或形成终结性场面。
- 成功骗出干扰并展开后，应立即获得巨大优势，而不是进入细碎的微小战力比较。

随机性应服务于刺激感，而不是制造无力感。随机牌序、战场特性和调度分支都应给玩家留下应对窗口、补救手段或高风险高回报选择。胜利应来自“看懂规则、抓住机会、承担风险并成功翻盘”的过程。

## 分层

- `app/config.py`
  统一管理配置、数据库路径、token 过期时间、密码策略等。
- `app/db.py`
  初始化 `peewee` 数据库连接与建表流程。
- `app/models.py`
  定义玩家、密码、token、构筑、房间和对局快照等模型。
- `app/dao.py`
  提供数据库读写方法，避免路由和服务层直接操作模型细节。
- `app/async_persistence.py`
  管理对局内快照的进程内最新态缓存与异步入库队列；只用于对局快照，不用于账号、构筑、登录态等关键资料。
- `app/auth.py`
  负责 token 创建、校验，以及 Flask 路由鉴权装饰器。
- `app/content/duel_common.py`
  内容层薄聚合入口。仅汇总 `app/content/common/` 与 `app/content/effects/` 中的定义；新增卡牌内容优先直接从 `app/content/effects/` 导入通用效果工具。
- `app/content/common/`
  维护异象对决内容层可复用的常量、卡牌工厂、区域操作、衍生牌定义等底层工具；不应反向依赖引擎服务层。
- `app/content/effects/`
  维护卡牌效果共用工具，按战斗、区域、标记、创生、目标、复制等用途拆分；卡牌文件应优先从这里导入效果能力，单张卡私有逻辑保留在对应卡牌定义文件。
- `app/content/duel_decks.py`
  维护异象对决的流派构筑。
- `app/content/items/`
  维护异象对决的异象道具卡牌定义；卡面名称和图标应优先使用 `app/static/images/item/` 中已有资源。
- `app/content/characters/`
  维护异象对决的异能者卡牌定义。异能者必须使用真实角色名、有效属性和有效立绘路径。
- `app/engine/events.py`
  定义通用事件常量。新增卡牌规则需要跨内容共享监听时，优先加明确事件，不要在主流程拼裸字符串。
- `app/engine/event_bus.py`
  统一执行事件队列；支持 `default handler`、`append`、`replace` 三类语义。
- `app/engine/application/`
  应用服务层与外部资源适配层。`build_service.py` 维护构筑保存、图鉴、catalog 等非对局内操作；`run_state.py` 维护对局快照加载、公开投影适配和持久化排队。可以读取内容定义和调用 DAO，但不承载回合推进、揭示结算、素材消耗等战斗规则。
- `app/engine/setup/`
  对局初始化层。维护战场配置、开局快照、套牌解析、初始手牌与牌库实例化；需要调用核心流程时使用显式回调对象，不从该层反向 import `game_service.py`。
- `app/engine/state/`
  状态契约层。维护 `JsonDict`、卡牌状态、阵营状态、地点状态、选择状态和 action payload 等 TypedDict/TypeAlias；运行时仍使用普通 JSON dict，避免影响快照存储和前端协议。
- `app/engine/flow/`
  回合流程层。维护规划同步、回合开始/结束、揭示顺序、素材消耗、撤退/胜负、undo checkpoint 等流程编排；可依赖 `rules`、`state` 和 `event_bus`，不得访问 Flask、路由、DAO、数据库模型或异步持久化。
- `app/engine/rules/`
  领域规则层。维护宣言、目标选择、素材匹配、战场状态、环合结算等可复用规则；规则函数应尽量保持纯规则计算或只修改传入快照，不访问 Flask、路由、异步持久化或数据库。
- `app/engine/ai/`
  AI 策略层。维护人机对手的部署、宣言和异能者行动决策；通过显式规则回调访问核心能力，不直接负责持久化或公开状态组装。
- `app/engine/projection/`
  状态投影层。负责把内部对局快照转换为面向某个玩家的公开 JSON，包括隐藏对手手牌、换算视角、日志文本替换、选择面板与战场公开信息；不承载规则结算和数据库写入。
- `app/engine/game_service.py`
  对局编排层。负责加载最新对局快照、校验玩家操作、推进回合、协调规则模块、派发事件并把结算后的快照交给异步持久化队列；不应继续吸收构筑/catalog、开局工厂、AI 策略、素材规则、宣言规则或公开状态投影等可独立分层的职责。
- `app/routes.py`
  提供 HTML 页面和 JSON API。只做参数校验、鉴权和调用服务。
- `app/templates/` + `app/static/`
  前端页面。前端不做规则计算，只根据后端状态渲染页面和发送操作；页面拆分为登录页、资料页、主界面、构筑页和对局页。
- `plugins/nte_account_db.py`
  插件侧独立数据库桥接层。直接连接同一个数据库文件，并使用相同表名进行数据交互，不引用 `app/` 下的业务代码。
- `plugins/nte_account.py`
  机器人桥接层，提供“注册账号”“重置密码”命令；只通过 `plugins/nte_account_db.py` 访问数据库。QQ 号作为账号，附带用户名时按最多 8 个汉字截断保存。
- `scripts/seed_mock_account.py`
  当前环境不运行机器人时，用于预置 mock 账号和密码，便于联调；该脚本直接写共享数据库，不参与网站主链路启动。

## 架构规则

- 路由层只做参数校验、鉴权和调用服务；不要在 `app/routes.py` 中写费用、抽牌、宣言、素材、胜负或 AI 决策。
- `app/engine/game_service.py` 是对局 API 编排入口，不是所有规则的堆放点。只保留路由直接调用的对局操作，加载/持久化/公开投影通过 `app/engine/application/run_state.py` 适配，新增可复用规则优先进入 `app/engine/rules/`，新增公开状态字段优先进入 `app/engine/projection/`，新增开局/套牌/战场初始化逻辑优先进入 `app/engine/setup/`，新增 AI 行为优先进入 `app/engine/ai/`。
- `app/engine/application/` 不写战斗规则；除 `run_state.py` 作为对局快照适配层外，应用服务不应依赖对局内临时快照、行动队列或揭示结算流程。
- `app/engine/flow/` 不访问 Flask、路由、异步持久化、数据库或 ORM 模型；需要加载快照、组装公开状态或排队持久化时，由 `app/engine/application/run_state.py` 或 `app/engine/game_service.py` 在上层完成。
- `app/engine/rules/` 不访问 Flask、路由、异步持久化或数据库；需要外部能力时通过参数、回调或显式规则对象传入。
- `app/engine/setup/` 和 `app/engine/projection/` 不反向 import `app/engine/game_service.py`；若必须复用核心流程能力，使用 `SnapshotFactoryRules`、`ProjectionRules` 这类显式回调对象。
- `app/engine/ai/` 不直接写库、不组装公开响应；AI 只决定动作，具体状态修改仍通过规则回调和快照完成。
- 内容层的卡牌效果优先写在对应 `app/content/items/*.py` 或 `app/content/characters/*.py` 中；通用效果工具进入 `app/content/effects/` 或 `app/content/common/`。主流程不得为单张卡长期增加硬编码分支。
- 宣言配置必须写在卡牌定义里；上层根据宣言配置统一处理“选择战场”“选择手牌”“选择墓地或牌库”等流程。墓地和牌库候选默认去重，并按通用牌库规则排序。
- 事件常量统一放在 `app/engine/events.py`。跨多张卡共享的时机加事件；单张卡私有逻辑优先留在内容文件中。
- 保持依赖方向稳定：`routes -> engine/game_service 或 engine/application -> engine/flow/setup/rules/ai/projection/event_bus -> engine/state/content`；DAO、数据库模型和异步持久化只允许在 `game_service` 或 `engine/application` 这类上层适配处使用。底层模块不应为了省事 import 上层模块。

## 数据流

1. 玩家向机器人 `Fairy` 发送 `注册账号` 或 `重置密码` 指令；QQ 号就是账号。
2. `plugins/nte_account.py` 通过 `plugins/nte_account_db.py` 直连共享数据库，创建或查找玩家并生成登录密码。
3. 玩家在浏览器登录页输入玩家号和密码。
4. 后端校验密码后签发 token，浏览器将 token 存储在本地。
5. 登录后玩家可在网页修改用户名和密码；用户名最多保留 8 个汉字。
6. 浏览器后续请求在 `Authorization: Bearer <token>` 中携带 token。
7. 路由层通过 `app/auth.py` 获取当前玩家，再调用服务层执行构筑、房间或对局逻辑。
8. 游戏状态以 JSON 快照方式保存。对局内操作先更新进程内最新态并返回，再异步入库；同一进程内的后续请求优先读取最新态，页面刷新后可读取最新态或最后一次已落库快照恢复。

## 插件边界

- `plugins/` 目录下的代码不能直接引用 `app/` 下的 Python 模块。
- 插件与网站后端之间只通过共享数据库表进行交互。
- 插件侧只维护注册和密码所需的最小数据库连接与表映射。

## 持久化策略

- 玩家信息、构筑、密码、token、房间、对局快照全部进入数据库。
- 账号、构筑、房间、登录态等非对局内资料必须同步入库成功后再返回，不能走异步队列。
- 对局内快照使用“进程内最新态缓存 + 异步入库队列”：用户操作完成规则结算和响应构造后即可返回，不等待 `GameRun.snapshot` 写库完成。
- 对局内后续请求必须优先读取进程内最新态，再回退到数据库，避免异步入库窗口内读到旧快照。
- 异步入库只适用于对局快照，不要扩散到注册、登录、构筑保存、房间成员变更等需要强一致的链路。
- 重置、重新开局、切换房间等会废弃旧对局的操作，必须让旧的待写快照失效，避免后台写回过期状态。
- 服务进程异常退出时，只保证恢复到最后一次已落库快照；这是对局内性能优化的取舍，不代表关键资料可以丢失。

## 异象对决基础规则

- 当前目标版本倾向单战场牌桌，只保留一个主战场，同时保留“战场特性 buff”制造每局差异。
- 如旧实现仍存在 3 个异象空间，新增设计和测试应优先朝单战场收敛，避免让玩家分摊精力或被格子容量卡住。
- 常规对局为 6 回合；每回合可用能量等于当前回合数，最高 6 点。
- 双方共享同一个回合流程。效果持续时间优先使用“本回合”和“持续一回合”，不写“己方回合”或“对手回合”。
- 每个回合开始时先比较双方当前总战力并锁定本回合结算先手，然后进入部署阶段。本回合结算先手一旦锁定，不会因之后部署、素材消耗、战力变化或揭示效果而改变。
- 部署阶段只允许支付费用、放置卡牌和完成宣言；卡牌效果不能在双方结束部署前结算。
- 异象道具统一使用“揭示”作为卡面结算时机，不再设计单独的“使用”词条。双方结束部署后先锁定本回合部署记录，再声明并消耗异能者素材，然后按回合开始时锁定的结算先手揭示道具。
- 结算先手由回合开始时的当前总战力决定；当前总战力更高的一方成为结算先手，打平时随机。战斗版图不保留顶部栏，双方当前总战力、实时领先方、本回合结算先手、手牌数和牌库厚度应分别集中显示在左/右侧栏，随机结果写入日志。
- 揭示顺序为结算先手方先按部署顺序揭示并结算自己的全部本回合部署道具，随后结算后手方按部署顺序揭示并结算自己的全部本回合部署道具。
- 需要检视牌库、二选一或选择目标的道具，必须在部署该牌时完成宣言并锁定选择；揭示阶段只实现已宣言的选择，不再重新选择。
- 类别检索默认由该牌操作者在合法候选中主动选择，候选列表只对操作者可见，最终影响的牌名公开并写入日志。若揭示时宣言对象不再合法，该部分不结算，不允许改选。
- 复制含有宣言目标的效果时，复制件不能弹出新的选择；卡面必须写明自动目标规则，没有合法自动目标时不复制该指向部分。
- 只有进入本回合前已经稳定在场的异象道具可以作为异能者素材；本回合部署中的道具、由本回合效果部署或生成的道具，都要等本回合完全结算后才稳定入场，不能被选为当回合素材。
- 揭示效果可以影响和检测本回合已经揭示并结算过的卡牌和事件；不能影响尚未揭示的待揭示道具，也不能回溯改变已经完成的素材消耗、宣言选择或已经结算完毕的效果。
- 所有影响战场卡牌的效果只能选择或影响表侧表示的卡牌，不能影响背面、部署中或尚未揭示的卡牌。因此，结算先手方影响对手战场卡牌时，只能影响对手进入本回合前已经表侧在场的卡牌；结算后手方影响对手战场卡牌时，可以影响对手本回合已经揭示并表侧在场的卡牌。
- 卡牌效果写“本回合”时，可以读取部署锁定记录、素材消耗记录、已经存在的持续标记，以及按结算先后手已经发生的本回合揭示结果；不能读取尚未揭示的卡牌效果。
- 回合开始补牌为双方各从牌库抽 1 张，不再进行牌库顶二选一。战斗版图必须显示双方牌库，牌库区需要根据牌库厚度有视觉差异；回合开始补牌和揭示阶段从牌库加入手牌都要播放牌库到手牌的抽牌动画。
- 揭示阶段中卡牌效果影响其他战场卡牌时，必须播放从效果来源到受影响卡牌的箭头动画。结算先手方的全部揭示完成后，才轮到结算后手方揭示。
- 异象道具基础战力必须大于 0。道具被减为非正战力时破碎，移动至墓地。
- 异能者拥有基础战力。异能者战力非正时返回异能者编队。
- 异能者在已经登场过的回合，仍可以继续消耗素材触发效果并吞噬战力；该效果统一称为“共鸣”或“再共鸣”，不要再叫“部署”。
- 异能者登场条件至少需要 2 个素材，王牌可以需要 3 个素材或附加条件。
- 成功登场异能者必须带来明显收益，如铺场、清场、强力中转、资源回收或终端爆发。

## 环合规则

- 基础环合只有 4 个：创生、延滞、浊燃、黯星。
- 自己产生的环合标记都记录在自己侧；延滞、浊燃、黯星仍照常影响对手。
- 统一使用“设置环合”描述环合生成，不使用“埋下”。
- 创生、浊燃、黯星基础持续 3 回合。
- 延滞基础持续 1 回合；哈索尔可以提升延滞持续时间。
- 盈蓄不是基础环合，不进入普通牌库。当己方同时存在创生与延滞时，生成 1 张 -1 费 0 战力的盈蓄牌加入手牌。盈蓄的 -1 费用返还是费用规则，不视为卡牌效果；它可以在部署阶段即时改变可用能量，但不会触发检索、部署、战力变化或其他效果。
- 失谐不是基础环合，不进入普通牌库。当黯星引爆时，若己方同时存在浊燃，则追加触发失谐结算，用于清理低战力目标、打断素材链或使非正战力异能者返回异能者编队。

## 属性与流派职责

- 灵属性用于联动创生，负责养成、回复、保护和大身材。
- 相属性用于联动延滞，负责控速、加税、素材封锁和延长延滞。
- 咒属性用于联动浊燃，负责持续压制、削战力和滚雪球。
- 魂属性用于联动黯星，负责倒计时、引爆和返场。
- 光属性是创生/延滞润滑件或盈蓄核心；浔例外，仍是创生主终端。
- 暗属性是浊燃/黯星润滑件或失谐核心；安魂曲同时兼顾浊燃体系和失谐体系。
- 基础环合只能由对应属性相关异能者设置：灵设置创生，相设置延滞，咒设置浊燃，魂设置黯星。浔是当前唯一明确的光属性创生终端例外。
- 光、暗和泛用异能者可以调度、保护、回收、修正素材或触发融合，但不能无条件跨属性设置基础环合。
- 九原是灵属性角色，只能设置或强化创生，不能设置延滞；在盈蓄队中负责回收和维持创生侧资源。
- 鉴定师是泛用角色，可以作为创生队人选，也可以进入延滞队；其环合来自消耗素材映射，消耗光属性 + 灵属性素材时设置创生，消耗光属性 + 相属性素材时设置延滞，不能脱离素材任意选择环合。
- 达芙蒂尔主要属于失谐队；进入黯星队只是因为当前黯星可用魂属性角色不足，作为暗属性借位角色负责桥接和失谐触发，不作为黯星主设置者。

## 流派与异能者编队基准

- 创生 = 光 + 灵。异能者编队基准：娜娜莉、浔、薄荷、九原；鉴定师可作为泛用人选替补进入创生队。核心体验是加身材、养大怪，保证己方关键单位不容易被小额伤害刮死；创生是入门套，翻看牌库较少，干扰较弱，但抗干扰和赖场能力强。
- 延滞 = 光 + 相。异能者编队基准：哈索尔、翳、鉴定师、卡厄斯。核心体验是拖延对手行动，异能者都以控制效果为主；延滞要能真正阻止或拖慢异能者登场，而不是只削一点战力。
- 浊燃 = 咒 + 暗。异能者编队基准：阿德勒、早雾、安魂曲、白藏。核心体验是节奏压制，持续建立优势并压制对手；浊燃通过连续压低素材和关键牌，在中期滚出节奏差。
- 黯星 = 暗 + 魂。异能者编队基准：法帝娅、哈尼娅、海月、达芙蒂尔；达芙蒂尔是失谐主力在黯星队的临时借位。核心体验是爆发返场；前期乏力，依靠设置黯星、保护延迟爆点和后期引爆完成一波返场。
- 盈蓄 = 创生 + 延滞融合，不是独立道具体系。异能者编队基准：小吱、埃德嘉、九原、翳；必须含 1 名相属性异能者和 1 名灵属性异能者。核心体验是费用魔术：每回合打出大量卡牌，同时靠延滞减少对手卡牌使用。
- 失谐 = 浊燃 + 黯星融合，不是独立道具体系。异能者编队基准：达芙蒂尔、哈尼娅、早雾、安魂曲；必须含 1 名魂属性异能者和 1 名咒属性异能者。核心体验是纯粹控制，期望把对手压到空场，再依靠后期高费异象道具取胜。
- 异能者必须拥有明确属性和有效图片路径，不允许无属性或路径缺失。

## 异象道具设计规范

- 异象道具必须使用 `app/static/images/item/` 中已有头像和真实名称；数量不足时，按 Everness 道具类别补充。
- 道具类别可以来自 Everness 类别，例如耗材、货币、钓鱼、食品、家具、滑翔翼、重要、商品、材料、音乐、其他、任务、食谱、车辆等。不要使用角色皮肤/时装作为异象道具卡。
- 异象道具以启动、调度、回收、小干扰为主，不直接低成本设置基础环合，避免跨套路污染。
- 道具效果必须使用明确字段，例如“费用 <=2”“灵属性”“材料类别”“从牌库加入手牌”“从牌库置入墓地”“从墓地返回手牌”，不使用“检索 xx 件”这类非字段描述。
- 不同异象道具的效果不能重复。展开链参考游戏王的定向调度、送墓、回收、特殊部署；体系节奏参考影之诗的低费启动、中期资源转化和高费终端。
- 一套主牌组目标约 20 张异象道具，展开件与干扰件大致各半；不同流派可根据定位调整。
- 部署和揭示次数多的流派应能通过异能者复用墓地、定向调度和再共鸣，把最终卡牌结算数加异能者效果次数推到 20 次以上。

## 前端约束

- 前端不计算费用、抽牌、调度、素材消耗、异能者条件、战力变化、环合持续、胜负或 AI 决策。
- 前端只做渲染、提示、选择、按钮禁用控制、动画衔接和 token 携带。
- 需要宣言的异象道具，前端应在玩家确认部署该牌前完成候选展示和选择锁定；揭示阶段只播放执行动画和结果，不再弹出新的选择。
- 前端展示目标选择和受影响反馈时，只能把表侧表示的战场卡牌标为合法目标；背面、部署中或尚未揭示的卡牌不能被高亮为可影响对象。
- 护盾应作为战场单位上的短持续状态展示，悬停时展示来源、持续时间和“不会受到伤害”的具体含义。
- 对局内接口优先保证“操作反馈先到”：出牌、唤醒异能者、选择素材、选择目标等请求发出后，应立刻锁定相关按钮或拖拽状态，并播放意图动画或加载动画等待后端返回。
- 固定 1 秒级的表现延迟只允许用于后端返回的播放队列，例如揭示、抽牌、箭头、破碎、生成和结算摘要等回放动作；前端玩家自主操作（拖拽、点击出牌、选择宣言、选择素材、选择目标、点击结束回合、切换面板）必须即时反馈，不能为了等待动画节奏而延迟响应。
- 加载动画应服务于无缝体验：不能出现空白等待或按钮可重复点击；后端返回时不要粗暴中断已开始的关键动画，应在当前阶段自然收束后再进入结算动画和状态刷新。
- 页面切换和状态刷新应尽量复用已有 DOM 状态或显示稳定骨架，不要在数据返回前清空主要区域；能用局部更新解决的，不要整页重建。
- 构筑页和战斗区加载角色立绘时要控制首屏压力：为立绘容器设置稳定尺寸，避免布局跳动；当前选中角色可优先加载，其余角色立绘应延迟加载、按需解码或使用低成本占位。
- 属性图标必须通过稳定映射加载。没有有效属性的异能者属于数据错误；道具无属性时应显示类别图标或隐藏属性位，不允许出现 broken image。
- 战斗区异能者编队、大图详情和构筑页详情应使用同一套异能者大图与属性展示逻辑。

## 内容扩展规范

- 异象对决卡牌按类型维护在 `app/content/items/` 和 `app/content/characters/` 中，通用工具、衍生牌与构筑分别维护在 `app/content/common/`、`app/content/effects/` 和 `app/content/duel_decks.py`；`app/content/duel_common.py` 只作为薄聚合入口保留。
- 新增异能者时必须使用真实角色名、属性、头像、立绘和清晰素材条件；不要用临时虚构角色补位。
- 新增异象道具时必须使用 `app/static/images/item/` 里的真实图片和名称；缺资源时先补资源，再接入牌表。
- 内容定义中的效果描述应与实际字段一致，避免“xx件”“专属件”“盈蓄件”“失谐件”等玩家无法从规则中理解的说法。
- 主流程不应硬编码某张卡的特殊分支；能作为内容效果实现的规则应挂在卡牌定义或通用结算函数上。
- 跨多个回合或多个事件保留状态时，优先使用命名清楚的显式状态、对局快照字段或环合计数，不要写入不透明临时桶。

## 事件规则约束

- 事件常量统一定义在 `app/engine/events.py`，不要在主链路里直接拼裸字符串。
- 当某段规则需要被多个内容定义共享监听时，加新的 `GameEvent`。
- 当主流程里的一个时机已经稳定存在，但内容层拿不到它时，加新的 `GameEvent`。
- 如果只是单个内容私有的小逻辑，不要为它单独加事件，优先挂在已有事件上。
- 当某条规则需要“改写默认主链路”时，优先使用事件总线的 `replace` 语义，而不是在 `game_service.py` 增加长期 if/else 分支。
- 当某条规则需要跨多个事件保留状态时，优先注册显式运行时效果或在对局快照中维护命名清楚的卡牌状态。

## 验证与迭代

- 现行设计说明记录在 `docs/everness-item-chain-card-design.md`，实现和新增设计必须优先遵循该文件。
- 平衡迭代标准、固定配对、评分规则和浏览器验收流程记录在 `docs/duel-balance-iteration.md`。
- 早期规则和数值试算允许使用脚本预检查，但正式体验结论必须通过浏览器完成。
- 浏览器验收必须通过真实页面进入牌桌，使用 UI 完成出牌、选择素材、唤醒异能者、选择目标、结束回合和查看日志。
- 每轮迭代至少保留 6 场固定配对对局结果、每套卡组评分、最低分卡组问题归因、改动清单和下一轮验证重点。
- 调整时优先处理最低分卡组或系统性规则问题，避免同时大改导致因果不清。
