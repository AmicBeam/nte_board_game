# Shaft Buff Nanoka 校对验收清单

本文以网页角色数据与 Nanoka 1.2 数据为 Buff 配置验收口径。具体数值、持续时间、层数和触发条件以网页文本复核结果为准；实现状态以当前浏览器计算引擎 `app/modules/shaft/static/js/shaft_engine.js`、配置 `app/modules/shaft/static/data/buffs.json`、动作数据 `app/modules/shaft/static/data/actions.json` 为真值。仅出现在说明文档或 `awakenings.json` 中不算已实现。

校对来源：

- `https://everness.info/character`
- `https://nte.nanoka.cc/`
- `https://static.nanoka.cc/manifest.json`
- `https://static.nanoka.cc/nte/1.2/zh/weapon/{id}.json`
- `https://static.nanoka.cc/nte/1.2/zh/console.json`
- `https://static.nanoka.cc/nte/1.2/zh/character/{id}.json`

说明：

- 装备合法性是所有弧盘 Buff 的前置条件：角色类型必须匹配弧盘类型，非法组合不参与 Buff 与伤害验收。
- `Q` 对应「极轨终结」，`E` 对应「变轨技能」，`援护` 对应「援护技」。
- `action_hit` 按命中后触发，不影响本次命中，只影响后续结算点。
- 敌方 Debuff 统一挂在敌人身上：`延滞`、`黯星` 初始5秒；`浸染`、`覆纹` 持续12秒；`浊燃` 持续15秒。
- 角色工作表只导入基础攻击、生命、防御和角色元数据。所有角色固定以5%暴击/50%暴伤作为裸角色基线；角色被动和觉醒只有进入 `buffs.json` 才参与计算。
- 常驻角色 Buff 使用 `trigger.event = passive` 与 `duration.type = permanent`，默认不画在 Buff 线上，但会进入动作详情的生效增益。

## 当前引擎能力边界

当前 `buffs.json` 支持 `provider.kind == "character"`，以及 `passive`、动作开始、命中、结束四类触发。常驻被动在模拟开始时注册，可按觉醒门槛、角色、动作名/类型/标签、前后台、属性、敌方血量和敌方 Debuff 筛选目标；限时 Buff 支持持续、刷新、叠层、触发冷却和切人/指定动作清除。3觉及以上角色的普攻、E、Q、援护有效技能等级按通用规则 +1，但 `被动` 类型伤害不读取技能等级。

当前也支持环合触发事件、依赖其他 Buff 状态触发、共享状态键、按来源角色去重记录、触发时读取并消耗另一状态层数。动态效果可以读取当前负面状态数量、命名 Buff 层数、状态已持续时间和施加者基础属性。全队唯一 Buff 保留各装备者独立触发、独立冷却和独立时间线实例，实际计算时按唯一组只取当前生效数值最高的一份。敌方抗性已拆分为光、灵、咒、暗、魂、相和心灵七类；个人资源会显示在对应角色轨道上。

动作数据可以产生和消耗按角色槽位隔离的 `personal_resource`。安魂曲「噩梦」已改为敌方独立计时 Buff：最多10层，每层3秒，并按每层自己的1秒周期结算伤害；卡厄斯「罪状」仍只保留资源记账。

当前验收范围固定为单敌伤害轴，不模拟多敌人、击杀、敌方释放技能或角色受击。随机效果按期望处理。浔 Q 是普通0秒Q，不冻结轴时间、CD或 Buff 时间。追击、协攻、自动援护、创生追加、失谐强化、黯星追加和延滞结算等额外攻击由用户手工添加后台动作，引擎只负责条件警告、伤害和 Buff 结算。

以下内容不视为完整机制实现：

- `characters.json.modifiers` 中的历史增益；计算内核不再读取这些值。
- `actions.json` 中的追击、附着、DOT、心灵、黯星、延滞等标签；标签不会自动生成追击、协攻、环合或角色状态。
- `awakenings.json` 中的展示描述；未进入 `buffs.json` 或引擎通用规则就不参与计算。

## 完成标准

单个 Buff/机制只有同时满足以下条件才可移入文档末尾的已对齐部分：

1. 触发、目标、数值、持续、层数、冷却、清除条件和觉醒门槛均与校对来源一致。
2. 计算详情能区分 `triggered_buffs` 与 `applied_buffs`；资源和手工后台动作能显示来源、条件及前后值。
3. 至少覆盖正常触发、不满足条件、持续边界、叠层上限、切换/指定动作清除五类测试。
4. UI只展示状态，不自行计算角色规则；浏览器结果与同轴引擎结果一致。
5. 删除会造成重复计算的静态增益，并用固定轴回归总伤害、DPS、能量、环合和个人资源。

## 验收关注点

1. 验收数值、持续、层数和触发条件只看校对后规则。
2. 实现状态只用于标记差距，不作为规则正确性的来源。
3. 单层 Buff 的 `max_stacks` 应为1；明确多层的条目按规则配置。
4. `triggered_buffs` 显示当前动作触发的 Buff，`applied_buffs` 显示当前动作实际吃到的 Buff。
5. 非法角色与弧盘组合不作为验收依据。
