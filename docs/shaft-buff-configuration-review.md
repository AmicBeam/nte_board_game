# Shaft Buff Nanoka 校对验收清单

本文以 Nanoka 1.2 数据为 buff 配置验收口径。当前实现状态只用于标记“已对齐 / 部分实现 / 已登记但不可触发 / 待实现”，具体数值、持续时间、层数和触发条件均以 Nanoka 校对后的规则为准。状态以当前浏览器计算引擎 `app/modules/shaft/static/js/shaft_engine.js`、配置 `app/modules/shaft/static/data/buffs.json`、动作数据 `app/modules/shaft/static/data/actions.json` 为实现真值；仅出现在说明文档或 `awakenings.json` 中不算已实现。

校对来源：

- `https://nte.nanoka.cc/`
- `https://static.nanoka.cc/manifest.json`
- `https://static.nanoka.cc/nte/1.2/zh/weapon/{id}.json`
- `https://static.nanoka.cc/nte/1.2/zh/console.json`
- `https://static.nanoka.cc/nte/1.2/zh/character/{id}.json`

说明：

- 装备合法性是所有弧盘 buff 的前置条件：角色类型必须匹配弧盘类型，非匹配组合不能携带，也不参与 buff 与伤害验收。
- `Q` 对应站点「极轨终结」，`E` 对应「变轨技能」，`援护` 对应「援护技」。
- 持续时间单位为秒；最大层数通常为 1，只有 Nanoka 明确写出叠层时才大于 1。
- `action_hit` 按“命中后触发”处理，不影响本次命中，影响后续结算点。
- 敌方 debuff 统一挂在敌人身上：`延滞`、`黯星` 初始 5 秒；`浸染`、`覆纹` 持续 12 秒；`浊燃` 持续 15 秒。
- 真红部分动作通过「升腾之赤」获得 +30% 自增伤，Q1 本身也生效；真红激活 3 觉醒共鸣后，Q1 额外获得「真红3觉共鸣」+30% 自增伤，默认持续 13 秒，释放 Q2 或离开前台时提前失效。旧版 xlsx 动作行自增伤已从动作面板移入 `buffs.json`。

## 角色被动/觉醒

### 当前引擎能力边界

当前 `buffs.json` 已支持 `provider.kind == "character"`，并能按动作开始、命中、结束触发，处理持续时间、刷新/叠层、触发冷却、动作/前后台/元素/敌方状态筛选，以及离开前台或指定动作重置。觉醒等级也可作为触发条件；当前通用规则还会让 3 觉及以上角色的普攻、E、Q、援护有效技能等级 +1。

动作数据可以产生和消耗按角色槽位隔离的 `personal_resource`。安魂曲「噩梦」和卡厄斯「罪状」已经能随动作累计并显示在计算结果中，但目前没有资源上限、阈值校验、基于资源自动改写动作、后台派生动作条件校验或角色专属 UI，因此只能算“资源记账已实现”，不能算完整角色机制。

当前验收范围固定为单敌伤害轴，不模拟多敌人、击杀、敌方释放技能或角色受击。依赖这些场景的被动/觉醒记为“本阶段不支持”，不作为当前实现 blocker。浔 Q 按普通 0 秒 Q 处理，不冻结轴时间、CD 或 Buff 时间；“时停期间创生不暂停”等特殊时间规则留待后续单独设计。追击、协攻、自动援护等额外攻击不由引擎自动生成，统一由用户手动添加对应后台动作，引擎只负责校验条件、结算伤害和相关 Buff。

以下内容不视为完整机制实现：

- `characters.json` 中直接并入面板的常驻 `modifiers`。其中可能混有本应有触发条件的被动或觉醒，只能说明数值进入面板，不能证明条件、持续时间和作用范围正确。
- `actions.json` 中的 `追击`、`附着`、`DOT`、`心灵`、`黯星`、`延滞`等标签。标签能参与通用伤害分类和 Buff 条件，但不会自动生成追击、协攻、环合或角色状态机。
- `awakenings.json` 中的觉醒描述。当前主要用于页面展示，除明确写入 `buffs.json` 或引擎通用规则的项目外，不参与伤害结算。

### 未实现机制支持计划

计划按“先补引擎语义，再补简单配置，最后补自动行为”的顺序推进。每个阶段完成后都要增加引擎单测和至少一条浏览器轴验收用例，不能只更新 `buffs.json`。

#### P0：统一角色机制契约

1. 扩展个人资源：支持初始值、上限、最低需求、消耗失败、动作后资源快照和前端轨道展示；补齐当前未执行的 `personal_resource_required` 语义。
2. 增加角色状态与目标状态：支持命名状态、敌方标记、动作来源、开始/结束事件、层数、持续时间和指定动作清除。
3. 增加后台派生动作校验：追击、协攻、援护、清算等动作由用户手动添加，动作定义声明所需角色状态、前置动作、触发窗口与冷却；条件不满足时输出明确警告，不自动补动作。
4. 增加缺失事件：前台驻场计时、角色切换、治疗、护盾、环合触发/结束和削减倾陷值。不增加击败、敌方释放技能、受击、敌人数变化等本阶段范围外事件。
5. 把角色条件型数值从 `characters.json.modifiers` 迁入可审计规则；迁移前后为相同基准轴保存面板与伤害快照，避免重复计算。

#### P1：闭合当前四名部分实现角色

1. **真红**：保留已验收的升腾之赤和 3觉共鸣；补盈蓄状态、独行叠层、无视防御及 Q 命中降光抗，并校验 Q2/离场清除顺序。
2. **早雾**：为合法动作补浮空/压制标签，使已登记减防可触发；再补浊燃定向增伤，跳过击败相关觉醒。
3. **安魂曲**：定义噩梦上限、消耗和技能阈值；实现手工援护动作的切换来源匹配、5 秒触发冷却及援护入场增伤。
4. **卡厄斯**：定义罪状上限和消耗；实现追缉许可目标、许可目标增伤/无视防御以及 Q 后暴伤。

#### P2：只依赖现有事件或简单新条件的角色 Buff

优先接入白藏、阿德勒、薄荷、海月、哈索尔、翳和小吱。它们的大部分首批能力可表达为常驻、动作开始/命中、前台状态、敌方血量、敌人数、敌方 Debuff 或短持续叠层。此阶段同时清理薄荷、海月、翳、小吱等角色当前可能误作常驻的条件型 `modifiers`。

#### P3：依赖环合、目标标记或状态结束事件的机制

接入浔、九原、达芙蒂尔、法帝娅和哈尼娅：浔 Q 仅作为 0 秒 Q 设置命名状态，不改变时间流速；其余实现玫约目标、失谐与倾陷上限、黯星结束/生命汲取、合奏与王牌状态。先建立通用事件和状态，不在主循环里按角色名硬编码；九原的“目标释放技能”效果跳过。

#### P4：手工后台动作和队友联动

接入娜娜莉、伊洛伊和埃德嘉的追击、协攻、治疗、护盾与援护联动。追击、协攻和援护由用户手动添加后台动作，引擎只校验触发窗口、角色状态、冷却和 Buff；伊洛伊必须先补齐动作数据。主角在取得可核验来源前保持阻塞，不以猜测规则补齐。

### 完成标准

单个 Buff/机制只有同时满足以下条件才可从“部分实现/待实现”改为“已对齐”：

1. 触发、目标、数值、持续时间、层数、冷却、清除条件和觉醒门槛均与校对来源一致。
2. 计算详情能区分 `triggered_buffs` 与 `applied_buffs`；个人资源和手工添加的后台派生动作能显示来源、条件校验及前后值。
3. 至少覆盖正常触发、不满足条件、持续时间边界、叠层上限、切换/指定动作清除五类测试；不适用的类别需在测试中说明。
4. UI 能展示当前状态，但不自行计算角色规则；浏览器结果必须与同一轴的引擎结果一致。
5. 删除或迁移会造成重复计算的静态 `modifiers`，并用固定轴回归总伤害、DPS、能量、环合与个人资源。

## 验收关注点

1. 验收数值、持续时间、层数和触发条件只看 Nanoka 校对后规则。
2. 不把现有实现表当作正确答案；实现状态只用于标记差距和验收进度。
3. 单层 buff 的 `max_stacks` 应为 1；Nanoka 明确多层的条目按上表最大层数配置。
4. `triggered_buffs` 应显示当前动作触发出的 buff，`applied_buffs` 应显示当前动作实际吃到的 buff。
5. 弧盘必须满足角色类型匹配后才可携带和结算；非法角色 + 弧盘组合不作为验收依据。
